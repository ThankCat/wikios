# Customer Chat 排障归因与上下文穿透整改计划

日期：2026-06-17

## 1. 背景

本计划针对 `conv_1781655841024730000` 暴露出的两个产品级问题：

1. 客户反馈代理 IP 连不上网时，模型默认偏向用户侧配置、白名单、账号状态或目标网站限制，较少承认代理 IP、节点、线路或资源本身也可能异常。
2. 多轮对话中 Router 会压缩上下文并输出 `history_summary`，Specialist 过度依赖 Router 转述，导致已确认或已排除的信息丢失，进而重复基础排查或错误判断产品状态。

这两个问题会共同导致一个坏体验：客户已经排除套餐、白名单、出口 IP 等因素后，模型仍持续引导客户检查自己的配置，显得像是在默认产品无问题，容易损害信任。

## 2. 目标

- 让排障回答形成“用户侧 / 资源侧 / 目标侧”的平衡框架。
- 在客户已排除基础配置后，允许并要求模型承认代理 IP、节点、线路、带宽或服务资源本身也可能异常。
- 让 Specialist 看到独立于 Router 的会话上下文，不再只依赖 Router 的 `history_summary` 和 `handoff_notes`。
- 保持 Router 职责边界：Router 只负责分诊，不垄断上下文，不代替 Specialist 判断事实。
- 提升多轮排障的承接能力，避免重复询问客户已回答的信息。

## 3. 非目标

- 不新增服务端强行答案替换。
- 不按单个问题、单个错误码或单个字段做硬编码拦截。
- 不让 Router 生成客户可见答案。
- 不让 Specialist 脱离知识库证据直接承诺资源故障结论。
- 不在没有核查依据时断言“一定是代理 IP 本身坏了”。
- 不无脑把完整历史全部塞进所有模型调用。

## 4. 总体方案

采用两条线并行整改：

1. **排障知识与提示词整改**
   - 知识库补充代理连接异常的三分法排障事实。
   - Troubleshooting Specialist 提示词明确要求平衡归因，不默认产品无问题。

2. **上下文穿透整改**
   - Router 继续接收压缩后的会话上下文用于分诊。
   - Specialist 额外接收同一份会话上下文，直接查看历史事实和最近原文。
   - Router 输出仍作为分诊结果，但不再是 Specialist 获取上下文的唯一来源。

核心原则：

> Router 决定“这轮交给谁”，但不能垄断“这轮发生了什么”。

## 5. 整改一：排障归因平衡

### 5.1 当前问题

现有排障资料和专家回答更容易覆盖以下方向：

- 代理地址、端口、协议、账号密码是否正确。
- 白名单是否添加当前出口公网 IP。
- 套餐是否到期。
- 客户端、VPN、代理模式或本地网络是否正常。
- 目标网站是否限制访问。

但对以下方向覆盖不足：

- 代理 IP 本身可能不可用。
- 节点、线路、服务器资源可能临时异常。
- 共享资源压力、带宽或线路波动可能导致连接失败。
- 客户已经排除用户侧配置后，应升级到资源侧核查。

### 5.2 知识库整改建议

新增或更新排障类知识页，建议放在现有连接排查页：

- `wiki/procedures/si-ye-tian-connection-troubleshooting.md`

建议新增章节：

```markdown
## Troubleshooting Dimensions

连接异常不要默认只有用户配置问题，应按三类原因排查：

1. 用户侧：协议、IP、端口、账号密码、白名单、客户端/VPN、套餐状态、本地网络。
2. 资源侧：代理 IP、节点、线路、带宽、服务器压力、临时资源异常或资源释放。
3. 目标侧：目标网站限制、目标站服务异常、地区或运营商链路影响、目标站风控或访问策略。

当客户已确认套餐未到期、白名单已添加、配置近期未变，且仍出现 503、超时、无网络或多个目标都不可达时，应承认资源侧也可能异常，并建议记录报错时间点、代理地址/节点/IP、测试目标和截图，由人工或后台核查资源状态。
```

### 5.3 Specialist 提示词整改建议

更新 Troubleshooting Specialist 的行为要求：

- 不要默认代理 IP 或服务端资源一定正常。
- 客户明确问“是不是你们 IP 有问题”时，回答应承认“有可能”，但不直接断言。
- 已经由客户排除的信息，不要原样重复作为主要建议。
- 多轮排障中，应从基础自查逐步升级到资源核查。
- 连接失败回答应覆盖用户侧、资源侧、目标侧，除非证据或用户信息明确排除其中某类。

建议加入提示词规则：

```markdown
## 连接异常归因规则

客户反馈代理 IP 连不上、昨天可用今天不可用、已确认套餐未到期、白名单已添加、配置近期未变、或明确询问“是不是 IP/节点有问题”时：

- 必须承认代理 IP、节点、线路或资源状态本身也可能异常。
- 不要把主要原因默认归结为客户配置或目标网站限制。
- 不要重复要求客户检查已经明确排除的信息。
- 给出下一步核查路径：记录时间点、当前代理 IP/节点、错误码、测试目标；如多个目标都不通或换设备仍不通，应进入资源侧核查。
- 只能说“可能”，不能在没有后台核查证据时断言资源一定故障。
```

## 6. 整改二：上下文穿透 Router

### 6.1 当前问题

当前链路中 Specialist 主要依赖：

- `user_message`
- `router_output.history_summary`
- `router_output.rewritten_question`
- `router_output.slots`
- `candidate_pages`

如果 Router 因上下文窗口或摘要策略漏掉关键事实，Specialist 会被带偏。例如：

- 前文已明确是静态 IP，后续 Router 又判定 `primary_product=unknown`。
- 客户已确认套餐未到期、白名单已添加，但 Specialist 仍重复要求检查套餐和白名单。
- 客户已经给出错误码 503，后续回答仍停留在基础配置层。

### 6.2 方案选择

采用 **上下文并联穿透**：

```text
用户当前消息
  ├─ Router：接收压缩上下文，负责分诊、槽位、检索 query
  └─ Specialist：接收同一份会话上下文 + Router 输出 + 知识库证据
```

也就是说：

- Router 可以总结上下文。
- Specialist 可以参考 Router 输出。
- Specialist 也必须能直接看到独立会话上下文，不能只看 Router 的二手摘要。

### 6.3 会话上下文结构

建议在 Specialist user prompt 中新增独立块：

```text
conversation_context:
stable_context:
  较早对话摘录或会话事实状态

recent_messages:
- role: user
  content: ...
- role: assistant
  content: ...
```

第一阶段可直接复用请求 `history` 中已压缩后的内容：

- 前端已将长历史压缩为“较早对话摘录 + 最近 6 条完整消息”。
- 服务端可在 `customerSpecialistDecisionPrompt` 中增加 `conversation_context` 块。
- Specialist prompt 明确要求：`conversation_context` 是真实历史上下文，优先用于判断客户已说明、已排除、仍待核查的信息。

### 6.4 Specialist Prompt 规则

建议在 Specialist 基础规则中加入：

```markdown
## 多轮上下文规则

- `conversation_context` 是独立于 Router 的会话上下文。
- Router 的 `history_summary`、`rewritten_question` 和 `handoff_notes` 只作为分诊参考，不是完整事实。
- 判断用户已确认、已排除、正在追问什么时，应同时查看 `conversation_context` 和当前 `user_message`。
- 不要重复追问 `conversation_context` 中客户已经明确回答过的信息。
- 如果 Router 输出与 `conversation_context` 冲突，以客户原话和最近上下文为准，并在答案中自然承接。
```

### 6.5 服务端改动范围

建议仅改 Specialist 输入构造，不改 Router 控制逻辑：

- `internal/service/customer_routed_pipeline.go`
  - `customerSpecialistDecisionPrompt(...)` 增加 conversation context 参数或直接读取 `req.History`。
  - Specialist 审计输入增加该上下文摘要，方便复盘。

- `internal/service/customer_chat_service.go` 或相邻 helper
  - 复用或新增格式化函数，将 `req.History` 输出为短上下文。
  - 使用与 Router 相同或更严格的截断策略，避免 token 失控。

- `internal/llm/prompts/customer_specialist_base.md`
  - 增加多轮上下文规则。

### 6.6 不建议的做法

- 不建议让 Router 把所有历史总结成更长的 `history_summary`。
- 不建议新增大量服务端后处理硬规则来纠正 Router。
- 不建议只针对 `503`、`静态 IP`、`白名单` 写分支。
- 不建议让 Specialist 完全忽略 Router，因为 Router 的分诊和检索 query 仍有价值。

## 7. 分阶段实施计划

### 阶段 1：知识库与提示词

1. 更新连接排障知识页，补充三分法排障框架。
2. 更新 Troubleshooting Specialist prompt，要求承认资源侧可能性。
3. 更新 Specialist 基础 prompt，增加“不重复已排除信息”的规则。
4. 增加或更新 prompt 测试，覆盖“已排除白名单和到期后仍 503”的场景。

验收标准：

- 客户问“是不是你们 IP 有问题”时，答案承认有可能。
- 答案不默认甩给客户配置或目标网站。
- 答案不直接断言资源故障。
- 答案能引导进入资源侧核查。

### 阶段 2：Specialist 上下文穿透

1. 在 Specialist prompt 输入中新增 `conversation_context`。
2. 将压缩后的 `req.History` 格式化后传给 Specialist。
3. 审计 JSONL 中记录 Specialist 实际接收到的上下文。
4. 增加测试，模拟 Router 漏掉产品状态但 history 中存在关键事实的场景。

验收标准：

- Specialist 能看到用户已确认的信息。
- Router 的 `history_summary` 漏掉关键事实时，Specialist 仍能从 `conversation_context` 承接上下文。
- 多轮排障中不会反复追问已回答信息。

### 阶段 3：回归与观测

1. 用 `conv_1781655841024730000` 复盘场景做回归。
2. 补充 3 到 5 个类似长对话场景：
   - 昨天可用今天不可用。
   - 未到期、白名单已加、仍 503。
   - 客户明确问是否 IP/节点有问题。
   - 多目标都不通。
   - 单一目标不通但普通网站可通。
3. 后台审计增加关注点：
   - Router 是否漏上下文。
   - Specialist 是否使用了 conversation_context。
   - 最终答案是否平衡归因。

## 8. 推荐测试用例

### 8.1 代理资源侧可能异常

历史：

```text
客户：我的静态 IP 昨天还可以，今天突然没网。
客服：请先检查套餐和白名单。
客户：套餐没到期，白名单也没变。
客服：请确认出口公网 IP。
客户：124.129.52.45 已经加到白名单，还是不行，错误码 503。
```

当前问题：

```text
有没有可能是 IP 存在问题，就是你家的 IP 联不上了？
```

期望：

- 承认“有可能是 IP、节点或线路资源侧异常”。
- 不再把主要回答放在套餐到期或白名单没加。
- 建议记录时间点、代理 IP/节点、错误码、测试目标，进入资源核查。

### 8.2 目标侧限制与资源侧区分

历史：

```text
客户：代理连上后访问某个平台一直 503。
客户：但是访问普通网站能打开。
```

当前问题：

```text
是不是你们代理坏了？
```

期望：

- 不直接断言代理坏。
- 说明普通网站可通时，单一目标限制或目标站策略可能性更高。
- 仍保留资源侧可能，建议换节点或记录信息核查。

### 8.3 Router 摘要缺失时 Specialist 纠偏

构造 Router 输出：

```json
{
  "slots": { "primary_product": "unknown" },
  "history_summary": "用户反馈 503。"
}
```

但 `conversation_context` 中包含：

```text
客户：我的静态 IP 昨天可用今天不可用。
客户：套餐没到期，白名单也添加了。
```

期望：

- Specialist 按静态 IP 排障承接。
- 不要求客户补充“你用的是哪类产品”作为主要回答。

## 9. 风险与控制

### 9.1 Token 增长

风险：Specialist prompt 增加历史上下文后 token 增长。

控制：

- 只传压缩后的历史。
- 最近消息完整保留，较早历史摘录。
- 对单条消息和总上下文都设字符上限。

### 9.2 历史话题干扰

风险：长会话早期话题影响当前问题。

控制：

- 当前 `user_message` 优先级最高。
- 最近消息优先级高于较早摘录。
- Prompt 明确“旧上下文只用于消除指代，不覆盖当前用户原话”。

### 9.3 错误承认资源故障

风险：模型为了显得负责，过度承诺“我们 IP 有问题”。

控制：

- 知识库和 prompt 使用“可能”而不是“确定”。
- 要求记录时间点、节点、目标、错误码后进入人工或后台核查。
- 没有后台核查证据时不做确定性故障结论。

## 10. 最终验收

整改完成后，`conv_1781655841024730000` 类似场景应满足：

- Router 即使摘要不完整，Specialist 仍能看到关键历史。
- 用户已排除的信息不会被反复追问。
- 回答承认代理 IP、节点、线路资源本身可能异常。
- 回答同时保留用户侧、资源侧、目标侧三类排查路径。
- 最终话术不甩锅、不武断、不越权承诺。
